## 3. 用户管理（区块链DAPP标准）

### 核心概念

> **区块链DAPP标准**：用户通过钱包授权登录，无需传统注册流程。用户身份以钱包地址为唯一标识。

### 用户识别方式
| 传统Web应用 | 区块链DAPP |
|-------------|-----------|
| 用户名/邮箱 | 钱包地址 |
| 密码登录 | 钱包签名授权 |
| 用户ID | Wallet Address |
| 注册时间 | 首次链上交互时间 |

### 查看用户列表
- 展示所有同步到数据库的用户
- 支持按钱包地址搜索
- 支持按等级筛选
- 支持按是否合伙人筛选

### 同步用户（SYNC FROM CHAIN）

由于用户数据存储在区块链上，管理后台需要从链上同步用户数据：

**步骤：**
1. 点击右上角「SYNC FROM CHAIN」按钮
2. 在弹出框中输入钱包地址
3. 点击「SYNC」确认同步
4. 系统将从区块链获取用户数据并存储到数据库

**说明：**
- 首次同步会创建用户记录
- 后续同步会更新用户最新数据
- 用户所有链上操作（投资、奖励等）都通过钱包地址关联

### 用户详情
点击用户卡片可查看详细信息：
- **钱包地址**（主标识）
- 推荐人地址
- 当前等级（S1-S6）
- 累计投资额
- 团队投资额
- 能量值
- LP份额
- 是否合伙人

### 区块链用户信息（chain-info）
每个用户还包含链上信息：
- 合约余额
- 最后活跃区块
- 累计交易数
- 账户创建状态

### 子页面
- **入金记录**：该用户的链上投资历史
- **奖励记录**：该用户获得的各类奖励
- **提现记录**：该用户的链上提现历史
- **团队成员**：该用户的直接推荐人列表

## 7. 节点申请管理

### 功能说明
管理用户提交的节点申请，包括节点合伙人（Node Partner）和节点委托人（Node Delegate）申请。

### 申请类型
| 类型 | 说明 |
|------|------|
| node_partner | 节点合伙人 |
| node_delegate | 节点委托人 |

### 申请状态
| 状态 | 说明 |
|------|------|
| pending | 待审核 |
| approved | 已通过 |
| rejected | 已拒绝 |

### 查看申请列表
1. 进入「节点申请管理」页面
2. 默认显示所有状态的申请
3. 可通过顶部筛选标签快速筛选：
   - ALL：全部
   - PENDING：待审核
   - APPROVED：已通过
   - REJECTED：已拒绝

### 申请信息
每条申请包含以下信息：
- 申请人钱包地址
- 用户昵称
- 申请类型
- 申请理由
- 联系方式
- 累计投资额
- 团队人数
- 附件链接
- 申请时间

### 审核申请（待审核状态）

**步骤：**
1. 点击待审核的申请卡片展开详情
2. 查看申请信息及附件
3. 点击底部「REVIEW」按钮
4. 在弹出框中填写审核备注（可选）
5. 选择操作：
   - **APPROVE**：通过申请
   - **REJECT**：拒绝申请
6. 确认后系统自动更新状态

## API 接口参考

### 基础信息
- Base URL: `http://localhost:9091/api/v1`
- 所有请求需在 Header 中添加 `Content-Type: application/json`

### 用户管理 API（区块链DAPP标准）

#### 获取用户列表
```
GET /api/v1/users
```
Query 参数：
- `page`: 页码（默认1）
- `pageSize`: 每页数量（默认20）
- `search`: 钱包地址搜索
- `level`: 等级筛选

#### 同步用户（从链上）
```
POST /api/v1/users/sync
```
Body 参数：
- `wallet_address`: 钱包地址（必填，Solana格式）

#### 批量同步用户
```
POST /api/v1/users/sync/batch
```
Body 参数：
- `wallet_addresses`: 钱包地址数组（最多100个）

#### 获取用户链上信息
```
GET /api/v1/users/:address/chain-info
```

#### 获取用户详情
```
GET /api/v1/users/:address
```

#### 获取用户入金记录
```
GET /api/v1/users/:address/deposits
```

#### 获取用户奖励记录
```
GET /api/v1/users/:address/rewards
```

#### 获取用户提现记录
```
GET /api/v1/users/:address/withdrawals
```

#### 获取用户团队
```
GET /api/v1/users/:address/team
```

### 节点申请相关接口

#### 获取申请列表
```
GET /api/v1/node-applications
```
Query 参数：
- `page`: 页码（默认1）
- `pageSize`: 每页数量（默认20）
- `status`: 状态筛选（pending/approved/rejected）
- `apply_type`: 类型筛选（node_partner/node_delegate）

#### 获取统计数据
```
GET /api/v1/node-applications/stats
```

#### 创建申请
```
POST /api/v1/node-applications
```
Body 参数：
- `user_address`: 用户钱包地址（必填）
- `apply_type`: 申请类型（必填）
- `user_name`: 用户昵称
- `apply_reason`: 申请理由
- `contact_info`: 联系方式
- `total_invest`: 累计投资额
- `team_size`: 团队人数
- `attachment_url`: 附件链接

#### 审核申请
```
PUT /api/v1/node-applications/:id/review
```
Body 参数：
- `status`: 审核状态（approved/rejected）
- `reviewer_id`: 审核人ID
- `reviewer_notes`: 审核备注

## 区块链DAPP开发标准说明

### 用户身份标识
在传统Web应用中，用户通过用户名/邮箱+密码登录。而在区块链DAPP中：

1. **钱包地址作为用户ID**
   - 每个用户在链上有一个唯一的钱包地址
   - 地址格式：Solana使用base58编码（如 `7EcDhSYGxXyscszYEp35KHN8vvw3svAuLKTzXwCFLtV`）

2. **签名授权替代密码**
   - 用户通过钱包（如 Phantom、Solflare）进行签名授权
   - 无需注册，直接使用钱包交互

3. **数据存储策略**
   - **链上数据**：投资、奖励、质押等核心业务数据存储在智能合约中
   - **链下数据**：索引数据、缓存、用户偏好等可存储在数据库中
   - 管理后台从链上同步用户数据到本地数据库

### 典型用户流程

```
用户端DAPP                    管理后台                    区块链
    |                           |                          |
    |-- 1. 钱包连接 ------------>|                          |
    |                           |-- 2. 同步用户数据 ------>|
    |                           |<-- 返回用户信息 --------|
    |<-- 显示用户面板 -----------|                          |
    |                           |                          |
    |-- 3. 用户发起投资 ------>|                          |
    |                           |-- 记录交易到数据库 ------|
    |                           |                          |-- 4. 链上交易确认
    |<-- 5. 显示交易结果 -------|                          |
```

## 常见问题

### Q: 为什么需要同步用户？
A: 区块链DAPP中，用户数据主要存储在链上。管理后台需要同步这些数据以便进行管理和分析。

### Q: 如何确认用户的钱包地址是否正确？
A: 钱包地址格式为base58编码的32-44个字符。系统会进行格式校验。

### Q: 同步失败怎么办？
A: 检查钱包地址格式是否正确，或检查网络连接是否正常。

### Q: 用户收不到通知？
A: 当前版本不包含消息通知功能，需手动通知用户或通过链上交易确认。
